# 日本語訳評価

本章では、Sloom Studio の日本語ローカライゼーションの品質、網羅性、一貫性を評価します。評価は、現在の翻訳インベントリと観測された UI 文字列に基づいています。総合評価、カバレッジ統計、適切な翻訳例と問題のある翻訳例、日本語組版対応の考察、そして具体的な推奨事項を示します。

## 総合スコア

- **アプリクロームと中核 UI:** **B+**
- **データ駆動型 UI と専門レジストリ:** **C+**
- **日本語組版機能:** **A-**
- **総合:** **B**

日本語訳は、共有アプリケーションクロームにおいて機能的かつプロフェッショナルです。ユーザーは、メニュー、ダイアログ、設定、一般的なワークスペースコマンドを日本語で混乱なく操作できます。ただし、多くのデータレジストリ（プロバイダーオプション、ツールラベル、プリフライトステータス、テンプレート名など）が英語のままです。これにより、言語が混在した体験になり、完成度が低下すると同時に、日本語 UI を期待するユーザーにとって混乱を招く可能性があります。

## カバレッジ統計

| レジストリ | 総文字列数 | 翻訳済み | カバレッジ |
|----------|--------------|------------|----------|
| `i18n.ts` カタログ | 677 | 677 | 100% |
| `workspaceMenus.json` | 246 | 245 | 99.6% |
| `nodeCatalog` | ~83 | ~83 | ~100% |
| `paperUsabilityActions` | 73 | 73 | 100% |

中核カタログの高いカバレッジは評価に値します。ローカライゼーション作業が体系的に行われ、主要なラベルの漏れがほとんどなかったことを示しています。実際の欠如は、これらのカウント対象カタログではなく、ローカライゼーションパスに含まれていない多くのデータレジストリにあります。

## 適切な翻訳例

日本語ローカライゼーションは、デザイン、印刷、漫画制作の領域知識を必要とする部分で優れています。例:

| 英語 | 日本語 | 備考 |
|---------|----------|-------|
| Bleed | 裁ち落とし | 正しい DTP 用語。 |
| Spreads | 見開き | 自然で、すぐに理解できる。 |
| Leading | 行送り | カタカナ語より適切で、日本語レイアウトに相応しい。 |
| Tracking | 字送り | 同様に適切な選択。 |
| Ruby / Furigana | ルビ | 標準的な用語。 |
| Emphasis dots | 圏点 | 正しい組版用語。 |
| Tate-chu-yoko | 縦中横 | 正しく、期待通りの用語。 |
| Vertical writing | 縦書き | 正しい。 |
| Kinsoku shori | 禁則処理 | 正しく、日本語組成において重要。 |
| Binding direction | 右綴じ / 左綴じ | RTL と LTR の綴じを正しく区別している。 |

これらの例は、ローカライゼーションチームが日本の出版用語を理解していたことを示しています。マンガ、書籍デザイン、印刷に携わるネイティブ読者はこれらの用語を認識し、アプリケーションを信頼するでしょう。

## 不自然または不適切な翻訳例

いくつかの翻訳は、文字通りの機械翻訳や過度に直訳的なカタカナ借用語のように感じられ、自然な日本語の UI 文案ではありません。

| 英語 | 日本語 | 問題 |
|---------|----------|-------|
| Pretty | きれい | デザインや効果のラベルとしてはくだけすぎて曖昧。綺麗、美麗、または機能的な説明など、より具体的な用語が望ましい。 |
| Middle align | 中段 | 組版では通常、垂直方向の中央揃えを意味する。中央揃えまたは中央寄せの方が標準的。 |
| Bounding box | バウンディングボックス | 3D ソフトで一般的だが、2D レイアウトでは境界ボックスや枠の方が自然。 |
| Star item | アイテムにスター | 文法的に不自然。アイテムにスターを付けるまたはスター付きアイテムが望ましい。 |
| Estimator | エスティメーター | 理解はできるが、この文脈では見積やコスト見積の方が明確。 |
| Organic | オーガニック | 日本語では食品を連想させることが多い。デザインの文脈では有機的、または形状に応じた用語が望ましい。 |

これらの不自然な文字列は、主にデータレジストリやツールラベルに現れ、`i18n.ts` の中核カタログではありません。これらは使用を妨げるほどではありませんが、アプリケーションの完成度を低下させ、時々混乱を招きます。

## 未翻訳または英語のみのローカライゼーション例

多くの専門レジストリは、未だに英語が主体です。日本語を選択したユーザーでも、これらの領域で英語のラベルを見ることになります。

| レジストリ / 領域 | よく見られる英語の例 |
|-----------------|------------------------------|
| プロバイダーカタログオプション | モデル名、エンドポイントの説明、機能ラベル。 |
| 画像ノードテンプレート | "inpaint"、"outpaint"、"remove background" などの操作ラベル。 |
| 画像プロバイダー機能ヘルプカード | 英語の説明文とツールチップ。 |
| Paper プリフライトステータスラベル | "Missing font"、"Low resolution image"、"Overprint issue"。 |
| Paper ドキュメントガイドラベル | ガイド名とプリセットの説明。 |
| Paper 吹き出しプリセット | "Speech Bubble Round"、"Thought Cloud" などのプリセット名。 |
| Image Editor ツールラベル | 一部のツール名とオプションが英語のまま。 |
| コマンドパレットアプリエントリ | プロバイダー固有およびレジストリ駆動のエントリ。 |

このパターンは一貫しています。静的で手書きされた文字列は翻訳されていますが、動的でレジストリ駆動の文字列は翻訳されていません。これは、UI が JSON や TypeScript のデータファイルから構築されるアプリケーションでは一般的な状況ですが、真に完全なローカライゼーションには対処する必要があります。

## 日本語組版機能

Sloom Studio の日本語組版対応は強みです。以下の機能はローカライズされ、正しく動作します。

- **ルビ:** `漢字《かんじ》` の入力構文に対応し、レンダリングされる。
- **圏点:** `《《強調》》` の入力構文に対応し、選択可能なドットスタイルを持つ。
- **縦中横:** 縦書き内の水平方向の列に対応。
- **縦書き:** テキストフレームを垂直フローに切り替え可能。
- **禁則処理:** 日本語テキストに行頭行末禁止文字の処理が適用される。
- **右綴じ:** 書籍やマンガ向けに右から左への綴じに対応。

これらの機能は単に翻訳されているだけでなく、正しい動作を伴って実装されています。ローカライゼーションが最も強いのはこの領域です。

## 一貫性の問題

カバレッジ以外にも、ネイティブな校正者が対処すべき一貫性の問題があります。

### 借用語のスタイル

一部の用語ではカタカナ借用語を使用し、近くの用語では日本語の対応語を使用しています。例:

- 裁ち落としの隣にバウンディングボックスがあると違和感がある。
- 見積ベースの用語の隣にエスティメーターがあると違和感がある。

アプリケーションは借用語ポリシーを決めて一貫して適用すべきです。良い指針は:

- ツールや UI 概念には定着した借用語を使用する（ブラシ、レイヤー、ノード）。
- 日本語に対応する領域概念がある場合は、和語または漢語を使用する（裁ち落とし、禁則処理）。
- 日本語で別の一般的な意味を持つ借用語は避ける（オーガニック）。

### 動詞形と文法

一部の文字列は、常体と敬体の指示文を混ぜたり、動詞が期待される場所に名詞句を使用したりしています。例えば、「アイテムにスター」はアクションラベルとして動詞句が望ましい場所に名詞句を置いています。メニュー項目、ボタン、ツールチップが動詞、名詞、短い句のいずれを使用するか、スタイルガイドで定義すべきです。

### 大文字と句読点

英語のレジストリエントリは、しばしばタイトルケースまたはセンテンスケースを保持します。日本語 UI では、カタカナのスペーシング、半角と全角の句読点、括弧のスタイルの一貫性が重要です。ネイティブエディターがこれらを標準化すべきです。

## 推奨事項

1. **中核カタログのネイティブ校正**
   `i18n.ts`、`workspaceMenus.json`、`nodeCatalog` の翻訳はほぼ完全ですが、一貫性、レジスタ、借用語の選択に焦点を当てたネイティブ校正から利益を得ます。

2. **データレジストリのローカライズ**
   最大の欠如は JSON/TypeScript のデータレジストリにあります。プロバイダーオプション、画像ノードテンプレート、画像プロバイダー機能ヘルプカード、Paper プリフライトラベル、Paper ドキュメントガイドラベル、Paper 吹き出しプリセット、Image Editor ツールラベルに日本語版を追加できるローカライゼーション層を追加します。

3. **借用語スタイルガイドの適用**
   カタカナ借用語と和語/漢語のどちらを使用するかを決定します。決定事項を文書化し、すべてのレジストリに適用します。エスティメーター、オーガニック、バウンディングボックスなどの用語を監査します。

4. **文法とレジスタの修正**
   アクションが暗示される場所では、名詞句を適切な動詞に置き換えます。メニュー、ダイアログ、ツールチップ全体で敬語レベルを標準化します。

5. **i18n テストの拡張**
   すべてのレジストリ文字列に日本語エントリがあることを検証する自動テストを追加するか、少なくともフォールバックして失敗しないようにします。現在のテストは主要カタログをカバーしていますが、データレジストリにも拡張すべきです。

6. **翻訳者向けの状況スクリーンショット**
   各文字列がどこに表示されるかのスクリーンショットを翻訳者に提供します。不自然な翻訳の多くは、翻訳者が周囲の UI を見ていなかったために生じた可能性があります。

7. **ユーザーからのフィードバックループ**
   翻訳の問題を報告する軽量なアプリ内フィードバック機能を追加します。日本語ユーザーは、裁ち落とし対ブリードのようなニッチな用語の修正に最適な情報源です。

8. **用語の文書化**
   Sloom Studio 用の日本語用語集を作成します。DTP、動画、画像編集、AI 用語の推奨訳を含め、将来の翻訳の一貫性を保ちます。

## 結論

Sloom Studio の日本語ローカライゼーションは、日本語組版や漫画を扱うユーザーにとって日常使いに十分な完成度です。中核 UI はよく翻訳されており、日本語特有の機能は正しく命名・実装されています。主な弱点は、英語のままのデータレジストリが多いことと、少数の不自然または過度に直訳的な翻訳です。データレジストリのローカライズ、ネイティブ校正、一貫したスタイルガイドに焦点を当てたパスを経れば、総合評価は B から A- 以上に上昇する可能性があります。
